## 1. WCAG 2.2 — What's New

### 1.1 Overview
WCAG 2.2 (Web Content Accessibility Guidelines) became the W3C Recommendation in October 2023. It builds on WCAG 2.1 (2018) without changing conformance requirements — new Success Criteria are Level A and AA only.

### 1.2 New Success Criteria in WCAG 2.2

| Criterion | Level | Name | Description |
|-----------|-------|------|-------------|
| 2.4.11 | AA | Focus Not Obscured (Minimum) | When a component receives focus, it must not be entirely hidden by other content (e.g., sticky headers) |
| 2.4.12 | AA | Focus Not Obscured (Enhanced) | Focus must not be hidden by any author-hidden content at all |
| 2.4.13 | AAA | Focus Appearance | Visual focus indicator must have sufficient area and contrast |
| 2.5.7 | AA | Dragging Movements | Any functionality using drag must also be achievable with a single pointer without dragging |
| 2.5.8 | AA | Target Size (Minimum) | Clickable targets must be at least 24×24 CSS pixels |
| 3.2.6 | AAA | Consistent Help | Help mechanisms must appear in consistent locations across pages |
| 3.3.7 | A | Redundant Entry | Information already entered by the user must not need to be re-entered |
| 3.3.8 | AA | Accessible Authentication (Minimum) | Cognitive function tests must not rely solely on memory or social interaction |
| 3.3.9 | AAA | Accessible Authentication (Enhanced) | No cognitive tests required for authentication |

### 1.3 Focus Appearance (2.4.13)
This is the most impactful new criterion for developers. Key requirements:
- Minimum contrast ratio of 3:1 between focus indicator and adjacent background
- Area: at least a 2px border around the element OR a 4px area around the element's bounding box
- Does not apply if the focus indicator is removed via CSS `outline: none` without replacement

### 1.4 Target Size Minimum (2.5.8)
- Requires 24×24 CSS pixels minimum per target
- Exceptions: links within a sentence, inline links, disabled components, targets in a UI component with equivalent larger targets
- Touch targets should ideally be 44×44px (Apple HIG) / 48×48dp (Material Design)
- CSS `min-height: 24px; min-width: 24px;` on the interactive element itself

### 1.5 Dragging Movements (2.5.7)
Any drag-and-drop interaction must have a single-pointer alternative:
- Sliders, drag-and-drop file uploads, kanban boards, reorderable lists
- Solutions: dual handles, arrow buttons, right-click context menus, voice commands

## 2. Semantic HTML Foundations

### 2.1 Why Semantics Matter
Semantic HTML provides the default accessibility tree. Screen readers, switch access, and voice control all rely on it. Wrong elements = broken experience even with perfect ARIA.

### 2.2 Landmark Regions
Use native landmarks instead of ARIA landmarks:
```html
<header>        <!-- banner landmark -->
<nav>           <!-- navigation landmark -->
<main>          <!-- main landmark (one per page) -->
<aside>         <!-- complementary landmark -->
<footer>        <!-- contentinfo landmark -->
<form>          <!-- form landmark when with accessible name -->
<search>        <!-- search landmark (HTML5) -->
<section>       <!-- region landmark when has accessible name -->
```

### 2.3 Headings
- One `<h1>` per page (page title)
- Headings must be nested correctly (no skipping levels)
- Avoid skipping: h1 → h3 without h2 creates confusion for screen reader users navigating by headings
- Use `<h1>`–`<h6>` for structure, not for visual styling

### 2.4 Links vs Buttons
| Action | Element | When to use |
|--------|---------|-------------|
| Navigation | `<a href>` | Moves to a new URL, resource, or page anchor |
| Action | `<button>` | Triggers JavaScript, submits form, toggles state, opens modal |

Common mistake: `<a href="#" onclick="doSomething()">` — use `<button>` instead.

### 2.5 Lists
Screen reader users navigate by list. Always use `<ul>`, `<ol>`, `<dl>`:
- `<ul>` / `<ol>` for related items
- `<dl>` + `<dt>` / `<dd>` for definition/value pairs

### 2.6 Tables
- Use `<table>` only for tabular data, not layout
- Always include `<caption>` for tables
- Use `<thead>` and `<tbody>`
- For complex tables, use `scope` attribute or `th#id` + `headers` for cell association

## 3. ARIA Patterns & Pitfalls

### 3.1 First Rule of ARIA
> "If you can use a native HTML element or attribute with the semantics and behavior you require already built in, then do so."

### 3.2 Common ARIA Mistakes

**Never do these:**
```html
<!-- WRONG: role="button" on a div when <button> exists -->
<div role="button" onclick="submit()">Submit</div>

<!-- WRONG: aria-label that contradicts visible text -->
<button aria-label="Close">X</button>

<!-- WRONG: aria-hidden on focusable element -->
<a href="#" aria-hidden="true">Learn more</a>

<!-- WRONG: role on wrong element -->
<div role="navigation">  <!-- nav is the element, not the role -->
```

**Common ARIA pitfalls:**
- `aria-label` without visible fallback — if the label is empty and no text content exists, the element is nameless
- `aria-live` without initial content — announce changes, but don't expect initial state to be announced
- `aria-expanded` on non-interactive element — must be on a button or element with `role="button"`
- Overlapping `aria-describedby` — multiple IDs can cause confusing announcements

### 3.3 ARIA Authoring Practices Guide (APG) Patterns

The [APG](https://www.w3.org/WAI/ARIA/apg/) provides reference implementations for common widgets:

| Widget | Pattern | Key Requirements |
|--------|---------|-----------------|
| Accordion | disclosure | `aria-expanded`, `aria-controls`, one panel open at a time |
| Tabs | tabs | `role="tablist/tab/tabpanel"`, arrow key navigation, `aria-selected` |
| Combobox | combobox | `aria-expanded`, `aria-controls`, `aria-activedescendant` |
| Dialog | dialog | Focus trap, `aria-modal`, escape to close, return focus on close |
| Menu / Toolbar | menu | `role="menubar/menu/menuitem"`, arrow key nav |
| Tree View | tree | `role="treegroup/item"`, expand/collapse with Enter/Space |
| Data Table | grid | `role="grid/row/cell"`, arrow key nav, editable cells |
| Carousel | carousel | Previous/Next buttons, live region for current slide |
| Skip Link | skip link | First focusable element, "Skip to main content" |

### 3.4 Modal / Dialog Pattern
```html
<div role="dialog" aria-modal="true" aria-labelledby="dialog-title" aria-describedby="dialog-desc">
  <h2 id="dialog-title">Confirm Action</h2>
  <p id="dialog-desc">This action cannot be undone.</p>
  <button id="confirm-btn">Confirm</button>
  <button id="cancel-btn">Cancel</button>
</div>
```
JS requirements:
1. Move focus to modal on open (usually the first focusable element)
2. Trap focus within the modal
3. Return focus to trigger element on close
4. Close on Escape key
5. Close on backdrop click

### 3.5 Live Regions
Used to announce dynamic content changes:
- `aria-live="polite"` — announces after current task (use most)
- `aria-live="assertive"` — interrupts immediately (use sparingly)
- `aria-atomic="true"` — announces the whole region, not just changed part
- `aria-relevant="additions text"` — what to announce

```html
<div aria-live="polite" aria-atomic="true">
  <span id="error-msg"></span>
</div>
```

## 4. Keyboard Navigation

### 4.1 Core Keyboard Requirements (WCAG 2.1 Level A)
- All functionality operable by keyboard (2.1.1)
- No keyboard trap (2.1.2)
- Skip mechanisms available (2.4.1)
- Focus order logical (2.4.3)
- Focus visible (2.4.7)

### 4.2 Keyboard Navigation Patterns

**Tab order:**
- `tabindex="0"` — adds to natural tab order
- `tabindex="-1"` — programmatically focusable but not in tab order
- `tabindex=">0"` — anti-pattern, creates unpredictable tab order

**Arrow key navigation:**
Used within composite widgets (tabs, menus, trees, grids):
- Arrow Up/Down: navigate within a group
- Arrow Left/Right: navigate between groups (for tabs)
- Enter/Space: activate
- Escape: close/dismiss
- Home/End: first/last item in a group

### 4.3 Skip Links
```html
<body>
  <a href="#main-content" class="skip-link">Skip to main content</a>
  <nav><!-- navigation --></nav>
  <main id="main-content">
```
```css
.skip-link {
  position: absolute;
  top: -40px;
  left: 0;
  background: #000;
  color: #fff;
  padding: 8px 16px;
  z-index: 100;
}
.skip-link:focus {
  top: 0;
}
```

### 4.4 Roving Tabindex Pattern
Used for widget-like components where focus is managed by JS:
```javascript
container.addEventListener('keydown', (e) => {
  const items = [...container.querySelectorAll('[role="option"]')];
  const current = document.activeElement;
  const currentIdx = items.indexOf(current);
  let nextIdx;

  if (e.key === 'ArrowDown') {
    nextIdx = (currentIdx + 1) % items.length;
    items[currentIdx].setAttribute('tabindex', '-1');
    items[nextIdx].setAttribute('tabindex', '0');
    items[nextIdx].focus();
  }
});
```

## 5. Screen Reader Testing

### 5.1 Supported Screen Readers

| Screen Reader | Platforms | Browsers |
|---------------|-----------|----------|
| NVDA | Windows | Firefox (recommended), Chrome |
| JAWS | Windows | Chrome, Firefox, Edge |
| VoiceOver | macOS / iOS | Safari (macOS), Safari (iOS) |
| Narrator | Windows | Edge |
| TalkBack | Android | Chrome |

### 5.2 Testing Workflow
1. Disable your mouse
2. Navigate the page using only a keyboard
3. Test with at least one screen reader (NVDA + Firefox on Windows recommended)
4. Check: page titles, headings, landmarks, links, form labels, dynamic content

### 5.3 NVDA Quick Commands
- `Tab` / `Shift+Tab`: next/previous focusable element
- `h` / `Shift+h`: next/previous heading
- `k` / `Shift+k`: next/previous link
- `d` / `Shift+d`: next/previous landmark
- `m` / `Shift+m`: next/previous form field
- `Insert+Down`: read entire document
- `Insert+Space`: toggle focus/flat review mode
- `Insert+F`: format information

### 5.4 VoiceOver (macOS) Commands
- `Cmd+F5`: toggle VoiceOver
- `Tab`: next focusable element
- `Ctrl+Option+Arrow keys`: navigate by element type
- `Ctrl+Option+U`: open rotor (headings, links, landmarks)
- `Ctrl+Option+Cmd+J`: jump to main landmark
- `VoiceOver+Space`: activate button/link

### 5.5 What to Test
- Page title changes on navigation
- Form field labels and instructions
- Error messages associated with fields (`aria-describedby`)
- Dynamic content announcements (aria-live regions)
- Image alt text accuracy
- Table captions and headers
- Dialog focus management
- Hidden content (visually hidden vs aria-hidden)

### 5.6 Testing Hidden Content
```css
/* Visually hidden but accessible to screen readers */
.sr-only {
  position: absolute;
  width: 1px;
  height: 1px;
  padding: 0;
  margin: -1px;
  overflow: hidden;
  clip: rect(0, 0, 0, 0);
  white-space: nowrap;
  border: 0;
}

/* Hide visually AND from screen readers */
.hidden-visually-and-from-at {
  display: none;
}
```

## 6. Color & Contrast

### 6.1 Contrast Requirements (WCAG 2.1)

| Criterion | Level | Requirement |
|-----------|-------|-------------|
| 1.4.3 | AA | Text: 4.5:1 normal, 3:1 large text (18pt+, 14pt bold+) |
| 1.4.3 | AAA | Text: 7:1 normal, 4.5:1 large text |
| 1.4.11 | AA | Non-text UI components: 3:1 against adjacent color |

### 6.2 Contrast in WCAG 2.2
- 1.4.11 (Non-text Contrast) applies to: input borders, focus indicators, buttons, checkboxes, radio buttons, dropdowns, slider tracks, progress bars, graph/chart elements, icons
- 2.4.11/2.4.12 (Focus Not Obscured) ensures focus indicators are visible
- 2.4.13 (Focus Appearance) requires 3:1 contrast AND minimum area

### 6.3 Tools
- [WebAIM Contrast Checker](https://webaim.org/resources/contrastchecker/)
- [Accessible Colors](https://accessible-colors.com/)
- [Color Oracle](https://colororacle.org/) — color blindness simulation
- [Stark](https://www.getstark.co/) — Figma/Sketch plugin
- [axe DevTools](https://www.deque.com/axe/devtools/) — browser extension

### 6.4 Color Independence
Never use color as the only visual means of conveying information:
```html
<!-- WRONG: Color-only error indication -->
<input style="border: 1px solid red" aria-invalid="true">

<!-- RIGHT: Icon + text + color + aria-describedby -->
<label for="email">Email address</label>
<input id="email" type="email" aria-describedby="email-hint email-error"
       aria-invalid="true">
<span id="email-hint">We'll never share your email.</span>
<span id="email-error" role="alert">Please enter a valid email address.</span>
```

## 7. Focus Management

### 7.1 Focus Visibility
```css
/* Always show visible focus — never suppress without replacement */
:focus {
  outline: 2px solid #005fcc;
  outline-offset: 2px;
}

/* Enhanced for WCAG 2.2 2.4.13 Focus Appearance */
:focus-visible {
  outline: 3px solid #005fcc;
  outline-offset: 3px;
  /* For elements with small bounding box, use box-shadow for area */
  box-shadow: 0 0 0 4px #ffffff, 0 0 0 6px #005fcc;
}
```

### 7.2 Focus Traps
Essential for modals, drawers, and any overlay:
```javascript
function trapFocus(element) {
  const focusable = element.querySelectorAll(
    'button, [href], input, select, textarea, [tabindex]:not([tabindex="-1"])'
  );
  const firstFocusable = focusable[0];
  const lastFocusable = focusable[focusable.length - 1];

  element.addEventListener('keydown', (e) => {
    if (e.key !== 'Tab') return;

    if (e.shiftKey && document.activeElement === firstFocusable) {
      lastFocusable.focus();
      e.preventDefault();
    } else if (!e.shiftKey && document.activeElement === lastFocusable) {
      firstFocusable.focus();
      e.preventDefault();
    }
  });
}
```

### 7.3 Return Focus on Close
```javascript
const previouslyFocused = document.activeElement;
modal.show();
modal.addEventListener('close', () => {
  previouslyFocused.focus();
});
```

## 8. Motion & Vestibular

### 8.1 WCAG 2.3 (Animations from Interactions)
| Criterion | Level | Requirement |
|-----------|-------|-------------|
| 2.3.1 | A | Three flashes or below threshold (no seizure risk) |
| 2.3.2 | AAA | Pause, stop, hide for animations lasting >5 seconds |
| 2.3.3 | AAA | No motion animation triggered by interaction |

### 8.2 prefers-reduced-motion
```css
@media (prefers-reduced-motion: reduce) {
  *,
  *::before,
  *::after {
    animation-duration: 0.01ms !important;
    animation-iteration-count: 1 !important;
    transition-duration: 0.01ms !important;
    scroll-behavior: auto !important;
  }
}
```
```javascript
const prefersReducedMotion = window.matchMedia('(prefers-reduced-motion: reduce)');
if (prefersReducedMotion.matches) {
  // Disable parallax, auto-play animations
}
```

### 8.3 Auto-playing Media
- Videos with sound: must not autoplay >3 seconds without a mechanism to pause/stop/mute
- Provide visible controls

## 9. Forms & Validation

### 9.1 Form Labeling
```html
<!-- Always use explicit label association -->
<label for="username">Username</label>
<input id="username" type="text" name="username">

<!-- For fields without visible labels: -->
<label for="search" class="sr-only">Search</label>
<input id="search" type="search" placeholder="Search...">
```

### 9.2 Accessible Error Messages
```html
<input type="email"
       id="email"
       aria-describedby="email-hint email-error"
       aria-invalid="true"
       aria-required="true">
<span id="email-hint">Enter your work email.</span>
<span id="email-error" role="alert" aria-live="polite">
  Please enter a valid email address (e.g., name@company.com).
</span>
```

Key: `aria-describedby` links the field to both hint AND error. `role="alert"` or `aria-live` announces the error immediately.

### 9.3 Required Fields
```html
<label for="password">
  Password <span aria-hidden="true" style="color: red">*</span>
  <span class="sr-only">(required)</span>
</label>
<input id="password" type="password" aria-required="true">
```

### 9.4 Form Validation Best Practices
- Validate on blur (not on keystroke for every character)
- Show errors after the user has finished the field
- Use `aria-invalid="true"` on the field when invalid
- Focus the first error field on form submission
- `role="alert"` or `aria-live="assertive"` for error containers

|------|------|---------------|
| [axe DevTools](https://www.deque.com/axe/devtools/) | Browser extension | ~30-40% of WCAG issues |

### 10.2 Manual Audit Checklist
- [ ] Keyboard-only navigation (disable mouse)
- [ ] Screen reader test (NVDA + Firefox or VoiceOver + Safari)
- [ ] All images have appropriate alt text
- [ ] All form fields have visible labels
- [ ] Color contrast meets minimum ratios
- [ ] Focus is visible on all interactive elements
- [ ] Dynamic content is announced via live regions
- [ ] No keyboard traps
- [ ] Skip link present and functional
- [ ] Page language set (`<html lang="en">`)
- [ ] Headings are nested correctly
- [ ] Page titles are descriptive and unique

## 11. Inclusive Design Principles

### 11.1 The Curb Cut Effect
Designing for the most marginalized first creates solutions that benefit everyone.

### 11.2 Include Don't Exclude
- Captioned videos help deaf users, non-native speakers, users in noisy environments
- Audio descriptions help blind users and users who skim videos
- Plain language helps users with cognitive disabilities, non-native speakers

### 11.3 Respect User Preferences
```javascript
window.matchMedia('(prefers-reduced-motion: reduce)');
window.matchMedia('(prefers-color-scheme: dark)');
window.matchMedia('(prefers-contrast: more)');
```

### 11.4 Progressive Enhancement
Build core functionality without JS, enhance with JS:
- Forms work without JS (server-side validation)
- Navigation works without JS
- Critical actions don't require complex gestures
